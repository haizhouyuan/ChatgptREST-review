- **severity:** critical  
	**file:** kernel/artifact\_store.py  
	**line:** 48-57, 134-155  
	**title:** SQLite 未开启 autocommit 且未显式 commit，Artifact 元数据在进程退出/跨连接后会丢失  
	**problem:**`_get_conn()` 使用 `sqlite3.connect(...)` 默认 `isolation_level=''` （非 autocommit）， `store()` 里执行 `INSERT OR REPLACE` 后 **没有 `commit()`** 。这会导致：
	1. 同一连接内能读到写入（测试会“看起来正常”），但
	2. 连接关闭、进程重启、或其他线程/连接读取时，插入记录可能不可见或直接回滚。  
		**impact:** “ArtifactStore”作为持久层会变成 **伪持久** ：产物文件写到了磁盘，但 DB 元数据可能没落盘；重启后 `get()/list_by_task()` 可能返回 None/空，导致溯源和交付链断裂。  
		**evidence:**
	- `_get_conn()` ： `sqlite3.connect(... check_same_thread=False)` 未设置 `isolation_level=None` （48-57）
	- `store()` ：执行 `INSERT OR REPLACE INTO artifacts ...` 后直接返回，无 `conn.commit()` （134-155）  
		**fix:** 任选其一（推荐 A）：
	- A（推荐）：在 `_get_conn()` 设置 autocommit：
	- B：在所有写操作（至少 `store()` ）后显式 `conn.commit()` ；并在异常路径 `rollback()` 。  
		同时补一个测试： **写入后关闭连接/新建 ArtifactStore 实例** 仍能读到记录（见 C 部分）。

- **severity:** critical  
	**file:** kernel/artifact\_store.py  
	**line:** 62-75, 134-155, 220-248  
	**title:** 以 artifact\_id 为主键存 provenance（task\_id/step\_id/producer）+ INSERT OR REPLACE，会覆盖溯源并破坏“多任务引用同一内容”的语义  
	**problem:** 表结构把 `task_id/step_id/producer/created_at/metadata` 放在 `artifacts` 表里，而主键是 `artifact_id` （内容哈希）。当 **同一内容** 被不同 task/step 产出时， `store()` 的 `INSERT OR REPLACE` 会用最后一次写入覆盖前一次 provenance。  
	**impact:**
	- `list_by_task(taskA)` 可能 **找不到** 该 artifact（因为被 taskB 覆盖成 taskB）。
	- provenance 变成“最后写入者获胜”，不符合“不可变+可追溯”预期。
	- 与 DEVELOPMENT\_PLAN 中“content-addressed + provenance”的方向冲突（计划里强调溯源）。  
		**evidence:**
	- 表： `artifact_id TEXT PRIMARY KEY` 同时存 `task_id, step_id, producer, created_at, metadata` （62-75）
	- 写： `INSERT OR REPLACE INTO artifacts (...) VALUES (...)` （134-155）
	- 查： `list_by_task WHERE task_id = ?`（220-248）会被覆盖后的 task\_id 影响  
		**fix:** 必须先明确语义，然后选一种一致方案：
	- 方案 1（推荐，符合“内容去重 + 多次产出溯源”）：拆表
		- `artifacts` （内容维度）： `artifact_id(pk), content_type, content_path, security_label, created_at`
		- `artifact_productions` （产出维度）： `production_id(pk), artifact_id(fk), task_id, step_id, producer, created_at, metadata, evidence_refs`
		- `store()` 返回 `artifact_id + production_id` ， `list_by_task` 查 productions
	- 方案 2（如果必须“一次产出=一个 artifact”）：把 artifact\_id 改为 hash(content + task\_id + step\_id + producer) 或引入 `artifact_instance_id` ，但这样“内容去重”要靠单独字段做。  
		同时修测试：验证同一 content 存两次（不同 task）后，两个 task 都能 list 到对应记录。

## B. Plan Gaps（计划偏差/未完成项）

以下对照 DEVELOPMENT\_PLAN.md 的关键承诺点（按影响排序），列出当前实现的偏差/缺口（只列“会影响系统行为/验收”的）：

1. **“config.py：所有路径从环境变量读取，不硬编码；所有 capability 使用 config.py 获取路径”未达成**
	- 计划：DEVELOPMENT\_PLAN.md 强调路径统一配置（尤其 Phase 0 不硬编码）
	- 现状：EvidenceLoad 已用 config（capabilities/evidence\_load.py 17-29），但 DraftGenerate 仍硬编码（capabilities/draft\_generate.py 15-17, 76-83）。
2. **“ArtifactStore content-addressed + provenance + 不可变”落地不完整且存在数据丢失缺陷**
	- 现状缺陷 1：DB 不 commit 导致元数据不持久（A/critical 第 1 条）。
	- 现状缺陷 2：用 artifact\_id(pk) 存 provenance 会覆盖（A/critical 第 2 条）。
	- 这会直接违背计划里“可追溯、可重放、可审计”的目标。
3. **“一切产出皆 Artifact / artifact.produced 事件链”未实现（或实现路径与计划不一致）**
	- 现状：capabilities 直接写文件，PipelineRunner 不调用 ArtifactStore，也不 emit `artifact.produced` 。
	- 结果：EventLog 无法从事件流还原“产物-步骤-证据”关系（只能靠文件路径猜）。
4. **“pipeline runner 事件语义与可重放性”存在偏差**
	- step.started 过早 emit（A/high 第 4 条）会污染事件语义；
	- step.finished payload 丢 retry\_exhausted 信息（A/high 第 6 条）会影响重放诊断；
	- task 级 trace\_id 在 TaskManager 中不稳定（A/high 第 9 条）与“全链路追踪”目标不一致。
5. **“fail-closed 的策略门禁”在 DraftGenerate 的存储侧未实现**
	- 计划强调策略门禁；
	- 现状：门禁判定在写盘之后执行（A/critical 第 5 条），导致“判定失败但内容已落盘”。
6. **“tests 覆盖验收关键不变量”存在缺口**
	- 现状测试数量多，但未覆盖 ArtifactStore 的“跨连接持久性”和“多 provenance 不被覆盖”等关键不变量（见 C）。

## C. Test Gaps（应补的测试）

按“能抓住 critical/high 缺陷”的优先级给出应补测试点（每条都能直接写进 tests/）：

1. **ArtifactStore：写入后跨连接/重启仍可读**
	- 场景：store -> 新建 ArtifactStore（或显式 close conn）-> get/list\_by\_task
	- 断言：记录存在且一致
	- 目的：捕获“未 commit”类持久性缺陷（当前测试不会发现）。
2. **ArtifactStore：同一 content 被不同 task/step store 后，两个 task 都能查到 provenance**
	- 场景：相同 content，两次 store（task\_id 不同）
	- 断言： `list_by_task(taskA)` 与 `list_by_task(taskB)` 都包含对应 production
	- 目的：防止 `INSERT OR REPLACE` 覆盖溯源。
3. **EvidenceLoad：TimeoutExpired fallback 必须写出占位文件**
	- 场景：mock `subprocess.run` 抛 TimeoutExpired
	- 断言：返回 pack\_path 存在且包含“超时占位”提示。
4. **DraftGenerate：门禁失败时不应写出包含敏感内容的文件**
	- 场景：构造 evidence\_content 含 email/phone；门禁应失败
	- 断言：
		- 要么 output\_path 不存在
		- 要么存在但不含敏感片段（只含脱敏/占位）
5. **DraftGenerate：输出目录必须来自 config.planning\_root 或 config.data\_dir（而不是 **file** 推导）**
	- 场景：设置 `AIOS_PLANNING_ROOT` 到临时目录
	- 断言：output\_path 在该临时目录下。
6. **PipelineRunner：step.started 不应在 skipped / cap-not-found / validation-fail 的情况下发出（或 actor 必须为 system）**
	- 需要给 EventLog 做一个 in-memory stub 或用 sqlite 临时库做断言。
	- 目的：固定事件语义，避免后续统计口径混乱。
7. **PipelineRunner：step.finished 的 error 字段应包含 retry\_exhausted 信息**
	- 场景：mock capability 连续失败直到耗尽重试
	- 断言：event payload 的 error 包含 `[RETRY_EXHAUSTED ...]` 。
8. **PolicyEngine：未知 security\_label 必须拒绝/降级（fail-closed）**
	- 场景：security\_label="secrettt" + audience="external"
	- 断言：allowed=False（或等价）。
9. **Registry：auto\_discover\_capabilities 必须真正注册且可执行**
	- 把 `assert count >= 0` 改成严格断言（见 A/medium）。
10. **PipelineContext.resolve：深层路径语法要么支持要么明确报错**
- 若决定支持：测试 `{{step1.output.result}}` 能正确解析；
- 若决定不支持：resolve 遇到多段路径应抛出明确错误而不是静默空字符串。
