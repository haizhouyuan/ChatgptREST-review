降低自动化检测风险的核心在于 **“消除机器特征”** 和 **“模拟人类行为”**。基于对 `haizhouyuan/ChatgptREST` 仓库代码（特别是 `chatgpt_web_mcp/server.py` 和 `ops/chrome_start.sh`）的深入分析，以下是针对该项目的全面改进建议：

### 1. 浏览器指纹与启动环境 (Browser Environment)

这是最容易被检测的环节。目前的 `ops/chrome_start.sh` 虽然挂载了用户目录，但启动参数仍带有明显的自动化特征。

* **隐藏自动化标志 (关键)**：
目前的启动脚本未明确禁用 Chromium 的自动化控制标志。建议在 `ops/chrome_start.sh` 的 `CHROME_EXTRA_ARGS` 中强制添加以下参数，以防止 `navigator.webdriver` 属性泄露：
```bash
--disable-blink-features=AutomationControlled
--disable-infobars
--no-first-run
--password-store=basic

```


*引用来源： (chrome_start.sh)*
* **注入 Stealth 脚本**：
在 `server.py` 中，虽然使用了 Playwright 连接 CDP，但缺乏主动的特征抹除。建议在 `_open_chatgpt_page` 或连接 CDP 后，立即注入以下 JavaScript（参考 `playwright-stealth` 的逻辑）：
```python
await page.add_init_script("""
    Object.defineProperty(navigator, 'webdriver', {get: () => undefined});
    window.chrome = { runtime: {} };
    // 伪造插件列表，防止被判定为无头/自动化浏览器
    Object.defineProperty(navigator, 'plugins', {get: () => [1, 2, 3]}); 
""")

```


这能防止简单的 JS 检测脚本直接读取到 `navigator.webdriver = true`。
* **窗口尺寸随机化**：
代码中 `CHATGPT_VIEWPORT_WIDTH` 默认为 1280x720。这过于标准。建议在启动时或连接后，对视口大小增加轻微的随机偏移（例如 1285x729），模拟真实用户手动调整窗口的行为。

### 2. 行为模拟与拟人化 (Behavioral Simulation)

代码中已经包含了一些延迟逻辑 (`_delay_range_ms`)，但可以更深入：

* **输入延迟抖动 (Typing Jitter)**：
`server.py` 中的 `_type_question` 使用了 `delay=_type_delay_ms()`。
* **建议**：不要使用固定的 `25ms`。人类打字速度是变化的。建议重写输入逻辑，使每个按键之间的延迟服从正态分布（例如均值 100ms，方差 30ms），并偶尔模拟“停顿”（思考）。


* **鼠标轨迹模拟**：
目前的 `click()` 操作通常是瞬间触发的。
* **建议**：在点击元素前，先使用 `page.mouse.move(x, y)` 移动到目标元素位置。如果可能，使用贝塞尔曲线算法生成非直线的鼠标移动轨迹，因为直线移动是机器人的典型特征。


* **增加“无意义”交互**：
真实用户在等待回答时，会滚动页面、选中文字或移动鼠标。
* **建议**：在 `_wait_for_answer` 循环中，随机插入少量的 `page.mouse.wheel(0, 100)` 滚动操作，或者随机移动鼠标，打破“静止等待 -> 瞬间接收”的模式。



### 3. 网络与 IP 策略 (Network Strategy)

项目中提到了 `mihomo` (Clash)，说明使用了代理，但这往往是数据中心 (Datacenter) IP，极易被 Cloudflare 标记。

* **使用高质量住宅 IP (Residential Proxy)**：
这是最有效的单一手段。不要使用云服务器或公共 VPN 的 IP。
* **IP 轮换与会话绑定**：
如果更换 IP，必须同时清除 Cookies 或使用无痕模式（但这与保持登录状态冲突）。
* **策略**：保持 IP 与 `user_data_dir` (Chrome Profile) 的强绑定。如果 IP 变了但 Cookies 没变，极易触发 `Unusual activity`。



### 4. 流量与频率控制 (Rate Limiting)

`rate_limit.py` 和 `worker.py` 实现了基于数据库的频率限制。

* **增加“非工作时间” (Sleep Cycles)**：
机器人是 24/7 工作的，人类不是。
* **建议**：在 `worker.py` 中引入“睡眠模式”。例如，连续工作 2 小时后，强制进入 15-30 分钟的长暂停。或者在夜间降低请求频率。


* **长尾延迟**：
目前的 `CHATGPT_MIN_PROMPT_INTERVAL_SECONDS` 默认为 61 秒。
* **建议**：不要让这个值太固定。将其改为 `60 + random(0, 30)` 秒。如果检测到 `Unusual activity`，自动将冷却时间指数级增加（例如从 20 分钟增加到 2 小时）。



### 5. 账号养护 (Account Warm-up)

* **不要使用“干净”的浏览器**：
`ops/chrome_start.sh` 使用了持久化目录 `secrets/chrome-profile`。这是好的做法。
* **建议**：在这个 Profile 中，人工登录 Google 账号，访问一些其他正常网站（YouTube, GitHub, Google Search），留下正常的浏览历史和 Cache。这能显著提高 Cloudflare 信任分数（Trust Score）。



### 6. 异常处理优化

* **被动防御 Cloudflare**：
代码中已有 `_raise_if_chatgpt_blocked` 检测 Cloudflare 挑战。
* **建议**：当检测到 `verification/captcha` 时，除了抛出错误，应立即停止所有 worker 对该账号的访问至少 1 小时，避免因持续尝试访问被阻断页面而导致账号被永久封禁。



### 总结实施清单

1. **修改 `ops/chrome_start.sh**`：添加 `--disable-blink-features=AutomationControlled`。
2. **修改 `server.py**`：
* 注入反指纹 JS。
* 优化 `_type_question` 实现变速打字。
* 在等待回复时加入随机鼠标/滚动行为。


3. **运营层面**：确保 IP 为住宅类型，并定期人工操作该浏览器实例以“保活”信任度。
